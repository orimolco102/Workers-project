# Nginx בפרויקט — README

## למה הוספנו את ה-Nginx?

עד עכשיו האפליקציה רצה בפורטים `443:3000` ב-Docker, ולכן האפליקציה (Node) הייתה צריכה להריץ הכל בעצמה — כולל את כל מה שקשור לתעבורה נכנסת מבחוץ.

הוספנו את ה-Nginx כדי לתת שכבת הגנה לקונטיינרים, וגם כמה יתרונות נוספים. השכבה הזו יושבת **לפני** שכבת האפליקציה: היא מקבלת את כל התעבורה מבחוץ, מעבירה אותה פנימה לאפליקציה, ובסוף מחזירה את התגובה חזרה למשתמש.

### הוספת ה-Nginx ל-`docker-compose.prod.yml`

```yaml
nginx:
  image: nginx:stable-alpine
  restart: always
  ports:
    - "80:80"
  volumes:
    # הוספת ה-volume וכיוונו אל המקום שבו נמצאת הקונפיגורציה של הסרוויס.
    # שיוך הנתיב מהמיקום על המחשב שלנו, אל המיקום בתוך הקונטיינר.
    - ./nginx/nginx.conf:/etc/nginx/conf.d/default.conf:ro
  depends_on:
    - app
```

## יתרון נוסף: Load Balancing

עוד משהו ש-Nginx מאפשר לנו הוא שימוש ב-**Load Balancing**. זה נותן לנו את היכולת ליצור קונטיינרים חדשים ולהוריד אותם במידת הצורך, לפי מצב העומס על השרתים. אם יש שרת שמבקשים ממנו הרבה מידע, מהרבה מקומות, ה-Load Balancer ייצור בעצמו קונטיינרים נוספים — וכך לא ייווצר עומס על קונטיינר בודד אחד.

---

## TLS ולמה לא הוספנו אותו

**TLS** הוא פרוטוקול תקשורת שאחראי על אבטחת התעבורה באתרים שמשתמשים ב-HTTPS.

במקרה שלנו, אנו מקבלים את השירות הזה מ-Render, ולכן בפרויקט שלנו לא היינו צריכים להוסיף ל-Nginx את פורט `443` ואת קבצי `fullchain.pem` ו-`privkey.pem` — Render כבר מטפל בשכבת ה-TLS/HTTPS עבורנו.

## סיכום מהיר

| נושא | הסבר |
|---|---|
| **מה זה Nginx כאן** | Reverse Proxy שיושב לפני האפליקציה, מקבל תעבורה מבחוץ ומעביר אותה פנימה |
| **למה הוספנו אותו** | שכבת הגנה על הקונטיינרים, וניהול טוב יותר של התעבורה הנכנסת |
| **Load Balancing** | יכולת (עתידית) ליצור/להוריד קונטיינרים אוטומטית לפי עומס, כדי לפזר עומס בין כמה קונטיינרים |
| **TLS/HTTPS** | לא הוגדר ידנית ב-Nginx בפרויקט הזה, כי Render מספק את השירות הזה עבורנו |
