# למה צריך Healthcheck למונגו ב-Docker Compose

ה-Healthcheck קיים כדי לתמוך בפונקציונליות של התלות בקונטיינר של מונגו. מכיוון שאנחנו תלויים בכך שקונטיינר המונגו יעלה, ורק אז נרצה שהקונטיינר הראשי יעלה, אנו נרצה לחכות שמונגו יעלה ויהיה מוכן לשימוש — כלומר שנוכל לבצע בו שימוש. איך נעשה זאת? על ידי שימוש בפקודות ב-docker-compose.

ב-Compose אנו נרצה להגיד לו שיחכה עד שהסרוויס של מונגו יעלה וידווח שהוא "בריא":

```yaml
depends_on:
  mongodb:
    condition: service_healthy
```

בסרוויס של מונגו נבצע את הבדיקה:

```yaml
test: ["CMD", "mongosh", "--eval", "db.adminCommand('ping')"]
interval: 30s
timeout: 10s
retries: 3
start_period: 30s
```

כמו שאנו רואים, הוספנו קוד שמשתמש ב-CMD על מנת להריץ את הפקודה `db.adminCommand('ping')`. הפקודה שולחת פינג לקונטיינר של מונגו ומחכה לתגובה ממנו. בתחילת ההרצה היא מחכה 30 שניות, שבמהלכן היא לא תוציא הודעת שגיאה (`start_period`). אם עברו 30 שניות, היא תבצע את הפקודה כל 30 שניות (`interval`), ובכל פעם תחכה 10 שניות לתגובה (`timeout`). אם אחרי 3 ניסיונות (`retries`) היא לא מקבלת תשובה מהקונטיינר, אנו נקבל הודעה: `container as unhealthy`.

```
Container workers-project-mongodb-1 Healthy
app-1      | ◇ injected env (0) from .env // tip: ◈ encrypted .env [www.dotenvx.com]
app-1      | ◇ injected env (0) from .env // tip: ⌘ override existing { override: true }
app-1      | mongodb+srv://********:*******@oridb.cs4lu95.mongodb.net/Workers?appName=OriDB
app-1      | app is runnng 3000
app-1      | MongoDB connected secsessfuly
```

ההבדל בין זה שהקונטיינר רץ לבין זה שהוא מוכן, הוא שהקונטיינר יכול להיחשב כמוכן מבחינת זה שהוא עלה, אבל האפליקציה שרצה בתוכו אולי לא עלתה (כמו אצלנו — הקונטיינר יכול לרוץ, אבל יכול לקחת לו זמן לעלות בגלל גודל האימג'). אם ננסה להשתמש באפליקציה של קונטיינר שלא עלה, אנו ניתקל בבעיות שונות. קונטיינר שלא עלה לגמרי לא יכול לספק את המטרות שלו לגמרי, ולכן אנו נרצה לחכות שיעלה וגם שיהיה מוכן. ה-Healthcheck בודק לנו אם הוא מוכן לשימוש, וגם אם האפליקציה עלתה בפנים.
